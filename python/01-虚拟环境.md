### **venv**‌ 

Python 内置的 `venv` 模块创建，核心作用是实现‌**项目依赖隔离**‌，避免不同项目间的库版本冲突 

**.venv**‌ 是 Python 项目中用于存储‌**虚拟环境**‌的默认目录名称

全称：Virtual Environments and Packages 



### .venv 

一个‌**文件夹目录**‌，通常位于项目根目录下，包含独立的 Python 解释器副本、pip 工具及项目专用的第三方库 

#### **核心作用**‌

**依赖隔离**‌：为每个项目创建独立的运行空间，确保项目 A 依赖的库版本（如 `requests==2.20.0`）不会影响项目 B（如 `requests==2.31.0`） 

**环境纯净**‌：避免将项目依赖安装到系统全局 Python 环境中，防止污染系统环境且无需管理员权限即可安装包

**便于迁移**‌：配合 `requirements.txt` 文件，可轻松在其他机器上复现相同的项目环境





### 创建虚拟环境

```shell
 # 基本语法
python3 -m venv 环境名称

python -m venv .venv
```

.venv 是常见约定，环境名称是必要参数

以 `.` 开头表示隐藏文件夹，保持目录整洁。





### 创建后的目录结构

```
.venv/
├── bin/            # 在 Unix/Linux 系统上
│   ├── activate    # 激活脚本
│   ├── python      # 环境 Python 解释器
│   └── pip         # 环境的 pip
├── Scripts/        # 在 Windows 系统上
│   ├── activate    # 激活脚本
│   ├── python.exe  # 环境 Python 解释器
│   └── pip.exe     # 环境的 pip
└── Lib/            # 安装的第三方库
```



### 激活虚拟环境

激活后，当前终端的 python 和 pip 命令将自动指向该虚拟环境，所有安装操作均在隔离空间内进行。

```shell
# macOS / Linux
source .venv/bin/activate 

# Windows（CMD / PowerShell）
.venv\Scripts\activate


```



### 导出依赖文件

```
pip freeze > requirements.txt
```



### 从文件安装依赖

```
pip install -r requirements.txt
```



### 退出环境

```
deactivate
```



### 删除虚拟环境

虚拟环境本质上是一个普通目录，删除目录即彻底移除：

```shell
# Linux
rm -rf .venv
# Windows
rmdir /s /q .venv
```





### 致命陷阱与避坑指南



#### 陷阱 1：忘记激活环境

这是最常犯的错误。在未激活环境下安装的包会进入全局环境，导致虚拟环境失去意义

#### 陷阱 2：将虚拟环境提交到 Git

虚拟环境包含大量二进制文件和特定路径配置，体积大且不可移植

务必在 `.gitignore` 中添加：

```
.venv/
venv/
env/
*.pyc
__pycache__/
```



#### 陷阱 3：移动虚拟环境

虚拟环境中的许多脚本（如 `activate` 和 Shebang 行）包含**绝对路径**

如果项目移动了，删除旧的 `.venv`，在新位置重新创建并安装依赖。在新位置重新创建环境并 `pip install -r requirements.txt`



#### 陷阱 4：Windows PowerShell 激活失败

以管理员身份运行，运行 `Activate.ps1`





### 最佳实践总结

1. **一文一环境**：每个独立项目都必须有专属虚拟环境。
2. **命名规范**：统一使用 `.venv` 作为目录名，方便 IDE 自动识别。
3. **忽略提交**：确保 `.gitignore` 中包含虚拟环境目录。
4. **锁定版本**：`requirements.txt` 必须包含具体版本号（`==`）。
5. **定期清理**：删除不再维护的项目的虚拟环境，释放磁盘空间。
6. **自动化**：在 CI/CD 流程中自动创建环境并安装依赖，确保测试一致性
